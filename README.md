## 1. Контекст и бизнес-цель

Промышленное предприятие с численностью около **5000 сотрудников** и **10 проходными** использует ручную проверку личности на КПП: дежурный сканирует пластиковый пропуск и визуально сверяет лицо сотрудника с фотографией в профиле.

Цель проекта — спроектировать информационную систему с ML-компонентом, которая автоматизирует идентификацию личности при проходе на предприятие и снижает зависимость от пластиковых пропусков, паролей и ручной визуальной проверки.

Ключевое архитектурное решение проекта: использовать не обычное распознавание по 2D-фотографии, а **3D-биометрию лица с определением глубины и проверкой liveness / presentation attack detection (PAD)**. Это нужно, потому что простая 2D-система может быть обманута распечатанной фотографией, изображением на телефоне или видеозаписью.

---

## 2. Что меняет внедрение системы

| Было | Станет |
|---|---|
| Сотрудник прикладывает пластиковый пропуск | Сотрудник подходит к терминалу, система сама выполняет биометрическую проверку |
| Дежурный визуально сравнивает лицо с фото | ML-модель строит embedding лица и сравнивает его с биометрическими шаблонами |
| Риск человеческой ошибки и «похожих лиц» | Решение принимается по порогу сходства, liveness и правилам доступа |
| Возможна передача чужого пропуска | Пропуск как основной фактор устраняется; требуется живое лицо с корректной глубинной картой |
| События доступа фиксируются ограниченно | Формируется полный audit trail: попытка, результат, устройство, модель, score, причина отказа |
| Проверка зависит от внимательности дежурного | Дежурный участвует только в исключениях: низкая уверенность, отказ, техническая ошибка |

---

## 3. Почему ML улучшает процесс

ML дает ценность не из-за «замены охранника камерой», а за счет сочетания:

1. **Face recognition / face identification** — поиск сотрудника по биометрическому шаблону.
2. **Liveness / PAD** — защита от распечатанных фото, видео на телефоне и простых масок.
3. **Автоматизированного принятия решения** — единая политика доступа, одинаковая для всех КПП.
4. **Аудита и мониторинга** — анализ отказов, ложных срабатываний, деградации модели и проблем устройств.

## 4. Выбранный подход

### 4.1. Режим работы

Основной режим — **1:N identification**: система сопоставляет текущий биометрический шаблон с базой активных сотрудников.

Дополнительный fallback-режим — ручная проверка дежурным через рабочее место оператора. Он нужен для отказа оборудования, плохого качества съемки, сильного изменения внешности, спорных случаев с низкой уверенностью модели, сотрудников без завершенной регистрации биометрии, посетителей и подрядчиков.

### 4.2. Почему добавлена глубина лица

Обычная камера получает только 2D-изображение. Для промышленного контроля доступа этого недостаточно: атакующий может предъявить фотографию или экран телефона. Поэтому терминал должен использовать сенсорный модуль класса **RGB + IR + depth**: камера видимого диапазона, инфракрасная камера/подсветка и датчик глубины или structured light.

Это увеличивает стоимость терминала и требует первичной регистрации каждого сотрудника, но улучшение оправдано:

- registration scan занимает около **10 секунд на сотрудника**;
- 5000 сотрудников × 10 секунд = около **13,9 часов чистого времени сканирования**;
- при 10 проходных и проведении регистрации в течение нескольких дней нагрузка реалистична;
- безопасность повышается существенно, потому что атакующему уже недостаточно иметь фотографию сотрудника.

## 5. Бизнес-требования

| ID | Требование |
|---|---|
| BR-01 | Система должна автоматизировать идентификацию сотрудников на 10 проходных. |
| BR-02 | Система должна поддерживать базу около 5000 сотрудников с запасом роста до 10000. |
| BR-03 | Среднее время прохода одного сотрудника не должно заметно увеличиться относительно текущего процесса. |
| BR-04 | Система должна снижать риск прохода по чужому пропуску или поддельному фото. |
| BR-05 | Все события доступа должны логироваться для расследования инцидентов. |
| BR-06 | Должен быть fallback-процесс для ручной проверки дежурным. |
| BR-07 | Биометрические данные должны обрабатываться по принципам минимизации, шифрования и ограниченного доступа. |
| BR-08 | Система должна поддерживать отзыв/удаление биометрического шаблона сотрудника при увольнении или отказе от участия, если такой отказ предусмотрен политикой предприятия. |

| Категория | Требование |

| Latency | P95 времени принятия решения на КПП — до 1,5 секунды при штатной работе edge-узла. |

## 9. Методология ML

Задача относится к нескольким классам:

- **биометрическая идентификация** — поиск наиболее похожего сотрудника по embedding лица;
- **biometric verification** — проверка совпадения с выбранным кандидатом;
- **presentation attack detection / liveness detection** — защита от подделок;
- **risk scoring** — принятие решения по комбинации ML-score и бизнес-правил доступа.

Минимально необходимые данные:

1. Биометрические шаблоны сотрудников, полученные при controlled enrollment.
2. Метаданные сотрудника: статус, подразделение, права доступа, график.
3. Метаданные устройств КПП.
4. События прохода и отказов.
5. Набор тестовых попыток с известной разметкой для оценки качества.
6. Негативные примеры presentation attacks: фото, экран телефона, видео, частичные маски, плохое освещение.

## 10. ML-метрики и связь с бизнес-результатом

| Метрика | Что измеряет | Связь с бизнесом |
|---|---|---|
| FAR / FMR | Доля ложных допусков чужого человека | Прямой риск безопасности |
| FRR / FNMR | Доля ложных отказов легитимным сотрудникам | Очереди на проходной и недовольство сотрудников |
| TAR@FAR | Вероятность верного допуска при заданном уровне ложного допуска | Позволяет выбрать безопасный threshold |
| APCER | Доля атак, которые система ошибочно приняла за настоящего человека | Устойчивость к spoofing |
| BPCER | Доля настоящих пользователей, ошибочно отклоненных PAD-модулем | Удобство использования |
| P95 latency | Время ответа системы | Влияет на пропускную способность КПП |
| Manual override rate | Доля ручных решений | Показывает, насколько система реально автоматизировала процесс |
| Enrollment success rate | Доля сотрудников, успешно прошедших регистрацию | Готовность к rollout |

## 11. Пилот

### Объем пилота

- 2 проходные из 10.
- 500–1000 сотрудников.
- Срок: 4–6 недель.
- Режим: сначала shadow-mode без открытия турникета, затем controlled production.
- Дежурный сохраняется как fallback.

### Критерии успеха

| Критерий | Целевое значение для пилота |
|---|---|
| P95 latency | ≤ 1,5 секунды |
| Successful auto-decision rate | ≥ 95% штатных попыток |
| Manual override rate | ≤ 5% после настройки |
| Критические false accept | 0 подтвержденных случаев |
| Очередь на проходной | Не хуже текущего процесса |
| Enrollment completion | ≥ 95% выбранной пилотной группы |
| Device uptime | ≥ 99% в рабочее время пилота |

## 12. MVP и технический долг

### Входит в MVP

- Enrollment сотрудников.
- 3D/depth capture и liveness/PAD.
- 1:N identification по активным сотрудникам.
- Интеграция с одной СКУД.
- Edge inference на КПП.
- Центральное хранилище шаблонов и событий.
- Операторская панель для спорных случаев.
- Базовый monitoring: latency, availability, reject reasons.
- Версионирование модели и threshold-конфигурации.

### Остается техническим долгом

- Полноценный self-service портал сотрудника.
- Продвинутая аналитика очередей и capacity planning.
- Автоматическая адаптация threshold по КПП/времени суток.
- Поддержка нескольких вендоров биометрических терминалов.
- Автоматизированный retraining pipeline на основе drift-сигналов.
- Расширенная SIEM-интеграция и UEBA.
- Детальный privacy-preserving подход с хранением шаблонов только на персональных токенах.

## 13. Production-архитектура

Система строится как распределенная:

- на каждом КПП есть биометрические терминалы и edge-узел;
- edge-узел выполняет capture validation, liveness/PAD и первичный inference;
- центральный кластер хранит authoritative данные сотрудников, шаблоны, события, конфигурации и версии моделей;
- локальный кэш на КПП нужен для продолжения работы при кратковременной недоступности центрального контура;
- события доступа отправляются асинхронно через message broker в audit/event storage.

```text
RGB+IR+Depth Terminal -> Edge Inference Node -> Access Decision API -> СКУД / турникет
                                      |
                                      v
                         Central Template Vault / Event Log / Monitoring
```

Подробнее см. [архитектурный документ](docs/03-architecture.md) и [диаграмму архитектуры](diagrams/04-system-architecture.puml).

| Риск | Вероятность | Влияние | Митигирующие меры |
|---|---:|---:|---|
| Атака распечатанным фото / видео | Средняя | Высокое | Depth + IR + PAD, тесты presentation attacks |

## 15. Вывод

Улучшение с 3D/depth-биометрией и liveness/PAD следует включить в проект как базовое требование. Без него система на обычной 2D-фотографии будет архитектурно слабой для промышленного контроля доступа: она автоматизирует процесс, но не решает ключевую проблему spoofing-атак. Дополнительное время на enrollment приемлемо для масштаба 5000 сотрудников, особенно если проводить регистрацию параллельно на нескольких проходных в течение нескольких дней.
