# Бизнес-процессы AS-IS и TO-BE

## 1. AS-IS: текущий ручной процесс

Текущий процесс на КПП:
1. Сотрудник подходит к проходной.
2. Прикладывает или передает пластиковый пропуск.
3. СКУД считывает ID пропуска.
4. Дежурный открывает профиль сотрудника или видит фото из системы.
5. Дежурный визуально сравнивает лицо человека с фотографией.
6. Если пропуск валиден и лицо «похоже», дежурный разрешает проход.
7. Если есть сомнения, сотрудник проходит ручную проверку или получает отказ.


### Проблемы AS-IS

| Проблема | Последствие |
|---|---|
| Субъективная визуальная проверка | Риск ошибки дежурного |
| Передача пропуска другому лицу | Возможен несанкционированный проход |
| Усталость и нагрузка на дежурного | Качество проверки нестабильно |
| Ограниченный аудит | Трудно расследовать спорные события |
| Отсутствие anti-spoofing | Фотография в профиле не доказывает присутствие реального сотрудника |

## 2. TO-BE: автоматизированный процесс после внедрения

Новый процесс:

1. Сотрудник подходит к биометрическому терминалу у турникета.
2. Терминал получает RGB-изображение, IR-изображение и карту глубины лица.
3. Edge-узел проверяет качество кадра и liveness/PAD.
4. ML-сервис строит embedding лица.
5. Система ищет сотрудника в базе активных биометрических шаблонов.
6. Access Decision API проверяет права доступа, статус сотрудника, проходную, время и результат ML.
7. При положительном решении СКУД открывает турникет.
8. При низкой уверенности или ошибке дежурный получает задачу ручной проверки.
9. Все события записываются в audit/event storage.

## 3. Главное изменение процесса

| Компонент | Было | Стало |
|---|---|---|
| Проверка личности | Человек визуально сравнивает лицо и фото | ML-система сравнивает embedding + depth/liveness |
| Решение | Дежурный принимает решение вручную | Decision API применяет пороги и правила доступа |
| Роль дежурного | Основной участник процесса | Участник fallback-сценариев |
| Доказуемость | Ограниченный журнал СКУД | Детальный audit trail с model version и scores |
| Безопасность | Пропуск можно передать | Нужна биометрия живого человека |

---

## 4. Исключения и fallback

Fallback-процесс обязателен, потому что биометрическая система не должна блокировать предприятие при технических или граничных ситуациях.

Причины fallback:

- сотрудник еще не прошел enrollment;
- терминал не видит лицо из-за каски, маски, очков, загрязнения камеры;
- liveness score ниже порога;
- match score в серой зоне;
- связь с центральным контуром недоступна дольше допустимого окна;
- сотрудник является посетителем или подрядчиком;
- сотрудник оспаривает отказ.

Решение дежурного также должно логироваться: кто принял решение, когда, по какой причине и на основании каких данных.
