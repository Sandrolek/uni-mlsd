# MLSD-документ

## 1. Цели и предпосылки

### 1.1. Зачем идем в разработку продукта

Предприятие хочет повысить надежность контроля доступа и уменьшить зависимость от ручной проверки личности на КПП.

Цели:

- исключить сценарий прохода по чужому пропуску;
- снизить человеческий фактор;
- ускорить проверку в часы пик;
- получить полный аудит событий доступа;
- централизовать правила допуска;
- повысить безопасность без перехода на отпечатки пальцев.

### 1.2. Почему ML лучше текущего процесса

Текущий процесс основан на субъективной визуальной оценке. ML-система позволяет сравнивать лицо с шаблонами одинаковым алгоритмом, фиксировать confidence score, автоматически обнаруживать spoofing-признаки, строить статистику отказов и улучшать систему через MLOps-процесс.

### 1.3. Ограничения

| Ограничение | Влияние |
|---|---|
| Биометрические данные чувствительны | Нужны privacy-by-design, DPIA, минимизация хранения |
| Сотрудники могут менять внешность | Нужен re-enrollment или adaptive update policy |
| Производственная среда сложная | Каски, очки, освещение, грязь на камере |
| Очереди на входе недопустимы | Нужен низкий latency и edge inference |
| Нельзя полностью полагаться на ML | Нужен fallback к дежурному |
| 2D-фото небезопасно | Требуется depth/liveness/PAD |

## 2. Методология

### 2.1. Технический тип задачи

Система включает несколько ML-задач:

1. **Face detection** — найти лицо в кадре.
2. **Quality assessment** — оценить пригодность кадра.
3. **Face embedding extraction** — преобразовать лицо в вектор признаков.
4. **1:N identification** — найти ближайший шаблон сотрудника.
5. **1:1 verification** — подтвердить совпадение с кандидатом.
6. **Presentation attack detection** — определить, является ли предъявление лица настоящим.
7. **Decision scoring** — объединить ML-score и бизнес-правила.

### 2.2. Необходимые данные

Обязательные данные:

- controlled enrollment captures каждого сотрудника;
- depth/IR данные для шаблона;
- employee master data;
- access rights;
- checkpoint/device metadata;
- события прохода;
- размеченные тестовые данные для genuine/impostor attempts;
- тестовые presentation attacks.

Желательные данные: разные условия освещения, каски, очки, респираторы, повторные проходы в разные дни, данные по отказам и ручным override, synthetic или vendor-provided PAD test sets.

### 2.3. Что нельзя делать

- Нельзя обучать production-модель на сырых данных без юридического основания и governance.
- Нельзя хранить все фотографии проходов бессрочно.
- Нельзя использовать только 2D-распознавание как основную меру безопасности.
- Нельзя открывать турникет при сбое PAD.
- Нельзя скрывать от сотрудников факт обработки биометрии.

## 3. Метрики качества

### 3.1. Face recognition

| Метрика | Описание |
|---|---|
| FAR / FMR | Вероятность принять чужого за сотрудника |
| FRR / FNMR | Вероятность отказать настоящему сотруднику |
| TAR@FAR | Вероятность правильного допуска при заданном FAR |
| Rank-1 accuracy | Доля случаев, где правильный сотрудник первый в списке кандидатов |
| EER | Точка равенства FAR и FRR, используется для сравнения моделей |

### 3.2. PAD/liveness

| Метрика | Описание |
|---|---|
| APCER | Attack Presentation Classification Error Rate |
| BPCER | Bona Fide Presentation Classification Error Rate |
| ACER | Среднее между APCER и BPCER |
| PAD fail rate | Доля отказов по liveness в production |

### 3.3. Product metrics

| Метрика | Описание |
|---|---|
| Auto-pass rate | Доля проходов без ручного вмешательства |
| Manual review rate | Доля спорных случаев |
| Average / P95 decision latency | Время принятия решения |
| Queue time | Время ожидания на КПП |
| Device uptime | Доступность терминалов |
| Enrollment completion | Доля зарегистрированных сотрудников |

## 4. Риски анализа и моделирования

| Риск | Что делаем |
|---|---|
| Недостаточно данных с production-условий | Shadow-mode, сбор статистики качества, пилот на реальных КПП |
| Bias модели | Оценка метрик по группам, fairness review, fallback |
| Overfitting к тестовым spoofing-атакам | Тестировать разные PAI: фото, экран, видео, маски |
| Drift условий съемки | Monitoring качества кадров и reject reasons |
| Плохой enrollment | Контроль качества при регистрации, повторный scan |
| Неправильный threshold | Подбор threshold по целевому FAR и бизнес-допустимому FRR |
| Vendor black-box | Требовать отчеты качества, API для audit, PoC с несколькими поставщиками |

## 5. Подготовка пилота

### 5.1. Способ оценки пилота

Пилот проводится в два этапа.

#### Этап 1: shadow-mode

Терминал сканирует лицо и считает решение, но турникет открывается по старому процессу. Система не влияет на доступ. Сравниваются решения ML и дежурного.

#### Этап 2: controlled production

Система открывает турникет автоматически при уверенном решении. Спорные случаи уходят дежурному. Ведется ручная валидация части событий.

### 5.2. Что считаем успешным пилотом

Пилот успешен, если:

- нет подтвержденных критических false accept;
- P95 latency ≤ 1,5 секунды;
- не возникает рост очередей;
- не менее 95% пилотной группы успешно прошли enrollment;
- auto-decision rate ≥ 95%;
- manual override rate ≤ 5% после настройки;
- операторы подтверждают, что fallback-процесс понятен;
- security/privacy review не выявил блокирующих проблем.

### 5.3. Подготовка пилота

1. Выбрать 2 проходные.
2. Установить терминалы и edge-узлы.
3. Подключить тестовую интеграцию со СКУД.
4. Провести enrollment пилотной группы.
5. Настроить шаблоны уведомлений и инструкции.
6. Подготовить операторов КПП.
7. Запустить shadow-mode.
8. Подобрать threshold.
9. Запустить controlled production.
10. Сформировать отчет и план rollout.

## 6. MVP

### 6.1. Scope MVP

В MVP входят: enrollment, терминалы RGB+IR+Depth, liveness/PAD, 1:N identification, интеграция с СКУД, журналы событий, операторская панель, мониторинг latency и ошибок, управление версиями модели, базовый privacy/security control.

### 6.2. Out of scope MVP

Не входят: visitor management, mobile-приложение сотрудника, полноценная BI-аналитика, автоматический retraining, интеграция со всеми возможными СКУД, поддержка нескольких типов биометрии, zero-trust архитектура в полном объеме.

---

## 7. Production rollout

1. Исправить проблемы пилота.
2. Утвердить legal/privacy/security документы.
3. Провести enrollment всех сотрудников.
4. Раскатить терминалы на все 10 проходных.
5. Включить production mode поэтапно.
6. Настроить регулярную отчетность.
7. Ввести регламент re-enrollment и incident response.
8. Проводить периодические PAD-тесты.

---

## 8. Требования к production

| Показатель | Требование |
|---|---|
| Количество сотрудников | 5000, запас до 10000 |
| Количество КПП | 10 |
| P95 latency | ≤ 1,5 секунды |
| P99 latency | ≤ 3 секунды |
| Central API availability | 99,5% |
| Edge offline tolerance | До 5 минут штатно, затем manual mode |
| RPO событий | 0 при восстановлении связи |
| RTO edge-узла | ≤ 30 минут |
| Data encryption | Обязательно |
| Audit trail | Обязательно |
| Fallback process | Обязательно |

---

## 9. Риски production

| Риск | Митигирующие меры |
|---|---|
| Массовый отказ терминалов | Запасные устройства, ручной режим, мониторинг |
| Ошибка новой модели | Canary deployment, rollback, model registry |
| Утечка шаблонов | Шифрование, KMS/HSM, RBAC, audit |
| Низкое качество в плохом освещении | IR/depth, тестирование на КПП |
| Рост очередей | Нагрузочные тесты, расширение терминалов |
| Юридическая претензия сотрудника | DPIA, прозрачное информирование, fallback |
| Spoofing новой техникой | Регулярный PAD-red-team, обновление моделей |
