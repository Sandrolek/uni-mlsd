# Риски, безопасность, privacy и compliance

## 1. Почему 2D-распознавание недостаточно

Система распознавания лица по обычной фотографии уязвима для presentation attacks:

- распечатанное фото;
- фото на экране смартфона;
- видео лица;
- частичные маски;
- 3D-маски.

Поэтому в проекте используется терминал с RGB+IR+Depth и liveness/PAD. Система должна проверять не только сходство лица, но и то, что перед камерой находится живой человек с корректной глубинной структурой лица.

## 2. Security controls

| Контроль | Описание |
|---|---|
| mTLS | Взаимная аутентификация терминалов, edge и central |
| Device certificates | Каждый терминал имеет сертификат устройства |
| Signed artifacts | ML-модели и конфиги подписываются перед rollout |
| Encryption at rest | Шифрование шаблонов, captures, backup |
| Encryption in transit | TLS для всех соединений |
| KMS/HSM | Централизованное управление ключами |
| RBAC | Разделение ролей: operator, admin, ml-engineer, auditor |
| Audit logs | Логирование доступа к данным и решений операторов |
| Fail-closed | При критической ошибке автоматический проход запрещается |
| PAD testing | Регулярные проверки на spoofing-атаки |

---

## 3. Privacy-by-design

Принципы:

1. **Минимизация** — хранить только необходимые шаблоны и события.
2. **Разделение данных** — employee master data отдельно от biometric template vault.
3. **Ограничение raw images** — не хранить снимки штатных проходов постоянно.
4. **Прозрачность** — сотрудники должны знать, какие данные обрабатываются и зачем.
5. **Сроки хранения** — шаблоны и captures должны иметь retention policy.
6. **Контроль доступа** — UI не должен позволять выгружать biometric templates.
7. **Аудит** — все обращения к чувствительным данным логируются.
8. **Удаление** — при увольнении шаблон отзывается и удаляется по регламенту.

---

## 4. Compliance considerations

Биометрические данные для уникальной идентификации человека относятся к особо чувствительным данным. Перед production-внедрением нужен как минимум:

- DPIA / оценка воздействия на защиту данных;
- определение правового основания обработки;
- информирование сотрудников;
- политика хранения и удаления;
- регламент ручного fallback;
- оценка необходимости и пропорциональности;
- security review;
- vendor risk assessment;
- документированный процесс обработки инцидентов.

Для учебного проекта это фиксируется как архитектурное и организационное требование, а не как готовое юридическое заключение.

## 5. Риски и меры

| Риск | Влияние | Меры |
|---|---|---|
| Spoofing фото/видео | Несанкционированный доступ | Depth + IR + PAD |
| 3D-маска высокого качества | Несанкционированный доступ | PAD-red-team, multi-sensor, threshold, ручная проверка |
| Ложный отказ | Очереди, недовольство | Серые зоны, fallback, улучшение enrollment |
| Ложный допуск | Критический security incident | Консервативный threshold, расследование, мониторинг |
| Утечка шаблонов | Невосстановимый ущерб | Шифрование, KMS, RBAC, audit, segmentation |
| Bias | Дискриминационные эффекты | Fairness evaluation, мониторинг FRR по группам |
| Vendor lock-in | Дорогая миграция | Стандартизированные API, abstraction layer |
| Ошибка обновления модели | Массовые отказы | Canary, rollback, model registry |
| Нарушение privacy | Юридические последствия | DPIA, minimization, retention, transparency |

## 6. Incident response

Типовые инциденты: подозрение на проход чужого человека, массовый рост отказов, подозрение на spoofing, утечка данных, компрометация терминала, сбой модели после обновления.

Минимальный процесс:

1. Зафиксировать событие и заморозить связанные логи.
2. Определить затронутые КПП, устройства и версии моделей.
3. Перевести КПП в safe/manual mode при необходимости.
4. Провести анализ access_event, match_attempt, operator_override.
5. Отозвать скомпрометированные ключи/сертификаты/шаблоны.
6. Подготовить отчет и корректирующие меры.
7. Обновить правила, модель или процедуру enrollment.
